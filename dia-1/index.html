🚚 La Ruta del Sabor — Comandera de Food Truck

Apliquen todo lo que vieron hoy sobre manejo de eventos (más el v-model de la clase anterior) construyendo un producto real: una comandera de pedidos.

🎬 La historia

Su amiga Dani abrió un food truck y está feliz... hasta que llega la hora punta. Toma los pedidos en una libreta, se le caen los papeles, suma mal los totales y la fila crece.

Les pidió a ustedes —su equipo de desarrollo— que le construyan una comandera digital: una pantalla donde toque los productos, arme el pedido y vea el total al instante.

Esto no es un ejercicio de juguete: es exactamente cómo funciona un Punto de Venta (POS) real de cualquier local. Hoy construyen la primera versión.

👥 Cómo se reparten el trabajo — equipo de 2

Este es un trabajo en equipo de verdad: NO se turnan un teclado. Se dividen el proyecto en dos frentes y cada persona es dueña de su parte. Al final, integran las dos piezas y prueban juntos.

👤 Persona A — "Front de venta" (agregar y totalizar)
  • Que al hacer clic en el menú, el producto entre al pedido → Requisito 1
  • Que el total se calcule y se muestre → Requisito 2
  • El estado vacío del ticket → Requisito 3
En una frase: A hace que el pedido se llene y el total salte.
👤 Persona B — "Gestión del pedido" (editar y cobrar)
  • Botones + − ✕ para editar cada línea → Requisitos 4 y 5
  • Nombre del cliente con v-model → Requisito 6
  • Cobrar sin recargar → Requisito 7
En una frase: B hace que el pedido se pueda editar y cerrar.

Nivel 3 (notas y descuento): repártanse los requisitos 8 y 9 que queden, uno cada uno.

⚠️ El punto de integración (léanlo juntos ANTES de empezar): los dos trabajan sobre el mismo estado pedido. Para que sus piezas encajen, acuerden desde el inicio la forma de cada producto del pedido:
{ id, emoji, nombre, precio, cantidad }.
Si A guarda los productos con esa forma, los botones de B funcionarán sin tocar nada. Ese acuerdo es el trabajo en equipo.
Cómo dividir en la práctica: el componente es uno solo, así que acuerden las partes y trabajen sobre el mismo archivo (cada quien sus métodos y su parte del <template>), o en dos computadores y luego juntan las piezas. Lo importante: cada quien responde por su parte y al final integran y prueban el flujo completo entre los dos.

🎯 Qué van a construir

Una comandera con dos zonas: el menú (se toca para agregar) y el ticket (el pedido con su total).

┌─────────────────────────┬──────────────────────────┐
│   MENÚ (tocar productos) │   TICKET / PEDIDO        │
│                          │                          │
│  🍔 Hamburguesa  $5.500  │   Cliente: [__________]  │
│  🌭 Completo     $3.200  │   ────────────────────   │
│  🍕 Pizza        $6.800  │   🍔 Hamburguesa  x2  ✕  │
│  🥤 Bebida       $1.500  │   🥤 Bebida       x1  ✕  │
│  🍟 Papas        $2.900  │   ────────────────────   │
│  🌮 Taco         $4.100  │   TOTAL:        $12.500  │
│                          │   [ Cobrar y cerrar ]    │
└─────────────────────────┴──────────────────────────┘
En el código base (más abajo, listo para copiar) el menú ya está construido (úsenlo de ejemplo) y el CSS está listo. Ustedes construyen el ticket (el HTML del pedido, dentro del <template>) y toda la lógica en setup(): el estado, los métodos y los eventos. Sigan las pistas del propio código y los requisitos de aquí abajo.

✅ Requisitos

Van por niveles. Terminen el Nivel 1 completo antes de pasar al 2. No salten.

🟢 Nivel 1 — El pedido básico (obligatorio)

  1. 👤 A Agregar productos al ticket. Al hacer clic en una tarjeta del menú, ese producto entra al pedido. Si ya estaba, súmale 1 a su cantidad (no lo dupliques).
  2. 👤 A Mostrar el total. Debajo del ticket, calcula y muestra la suma total del pedido. Debe actualizarse solo cada vez que cambie el pedido.
  3. 👤 A Estado vacío. Si el ticket no tiene productos, muestra el mensaje "Aún no hay productos".

🟡 Nivel 2 — Editar el pedido

  1. 👤 B Aumentar y disminuir cantidad. Cada línea del ticket tiene botones + y −. Si la cantidad llega a 0, ese producto desaparece del ticket.
  2. 👤 B Eliminar una línea. El botón ✕ borra ese producto del pedido completo.
  3. 👤 B Nombre del cliente. Un campo de texto donde Dani anota el nombre. Úsalo con v-model.
Pista de eventos (para B): el botón ✕ está dentro de la fila. Si la fila también reacciona al clic, usa el modificador correcto para que el clic del ✕ no dispare también el de la fila. (Repasa .stop.)

🔴 Nivel 3 — Desafío (si les da el ritmo)

  1. 👤 B Cobrar sin recargar. El botón "Cobrar y cerrar" está dentro de un <form>. Al enviarlo debe mostrar un mensaje tipo "✅ Pedido de {cliente} cobrado: $XX" y vaciar el ticket — sin que la página se recargue. (Repasa @submit.prevent.)
  2. 👤 A Nota rápida con Enter. Un input "Nota para la cocina" (ej: "sin cebolla"). Al presionar Enter se agrega a una lista de notas visibles. (Repasa @keydown.enter.)
  3. 👤 A Descuento por tecla. Un botón "Aplicar descuento 10%" que solo funcione una vez por pedido. (Repasa .once.)

🧠 Conceptos que DEBEN aparecer en su solución

Es la lista de lo que aprendieron hoy. Que no falte ninguno del nivel que alcancen:

  • @click para agregar productos y para los botones + − ✕
  • Método manejador con argumento — ej: agregar(producto), quitar(id)
  • Manejador en línea — una expresión corta directa en el @click
  • .stop — para que el ✕ no dispare el clic de la fila
  • @submit.prevent — para cobrar sin recargar
  • @keydown.enter — para la nota de cocina
  • .once — para el descuento de una sola vez
  • v-model — para el nombre del cliente y la nota (repaso Lección 3)
  • Un valor derivado (computed) para el total

💡 Pistas técnicas — no es la solución, es el mapa

  • Cada producto del pedido necesita guardar su cantidad. Piensen: ¿un arreglo de objetos { id, nombre, precio, cantidad }?
  • Para "si ya está, suma 1; si no, agrégalo": busquen el producto en el arreglo con .find().
  • El total NO se guarda en una variable que actualizan a mano. Se calcula a partir del pedido con computed. Si lo actualizan manualmente, tarde o temprano se desincroniza. El dato manda, la vista lo refleja.
  • Los métodos van dentro de setup() y se retornan para que el template los vea.

🏁 ¿Cuándo está "listo"?

Su comandera está lista cuando pueden hacer esta demo sin tocar código:

  1. Tocar 3 productos distintos → aparecen en el ticket con cantidad 1 cada uno.
  2. Tocar dos veces el mismo → queda en cantidad 2.
  3. El total cuadra con la suma real.
  4. Bajar una cantidad a 0 → el producto desaparece.
  5. ✕ borra una línea sin seleccionar/alterar la fila.
  6. Escribir el nombre del cliente y "Cobrar" → mensaje correcto y ticket vacío, sin recargar.
Si los 6 pasos funcionan: felicitaciones, construyeron un POS de verdad. 🎉

🚀 Cómo poner a andar su componente

Este código es un componente, así que vive dentro de un proyecto Vue (con Vite). Pasos:

  1. En su proyecto Vue, creen un archivo nuevo, por ejemplo src/components/ComandaPos.vue.
  2. Presionen "Copiar código" aquí arriba y peguen todo dentro de ese archivo.
  3. Úsenlo desde App.vue: 
    <template>
  <ComandaPos />
</template>

<script>
import ComandaPos from './components/ComandaPos.vue'
export default { components: { ComandaPos } }
</script>
  4. Levanten el proyecto con npm run dev y abran la URL que les indica la terminal.
  5. Cada vez que guarden, Vite recarga solo. Empiecen a construir el ticket. 🔨
Ojo: un componente .vue no se abre con doble clic en el navegador — necesita el proyecto Vite corriendo. Lo que sí se publica en GitHub Pages es esta página de instrucciones, para que puedan copiar el código desde cualquier lugar.
