💸 FinTrack — Control de Gastos en Equipo

Construyan entre dos personas una app real de finanzas hecha con varios componentes Vue que se comunican por props (datos que bajan) y eventos (avisos que suben). Cada uno programa sus propios componentes, y al final unifican todo en un mismo repositorio Git. Regla de oro del día: los datos bajan por props, los avisos suben por eventos.

🎬 La historia

Cata terminó el mes sin saber en qué se le fue la plata. "Creo que gasté como $200 lucas… ¿o eran $400?". No tiene idea. Anota algunos gastos en el bloc de notas del teléfono, otros los olvida, y nunca suma nada.

Les pidió a ustedes —su equipo de desarrollo— una app para registrar gastos: que ella escriba cuánto gastó y en qué, que la app sume el total sola y le muestre en qué categoría se le va la plata.

Esto no es un ejercicio de juguete: es exactamente cómo arranca cualquier app de finanzas (Fintonic, Mobills, Splitwise). Hoy construyen la primera versión… y la construyen como un equipo real, con el código repartido y unido por Git.

🎯 Qué van a construir

Una app con dos zonas: la izquierda para registrar y listar gastos, la derecha con el resumen que se calcula solo.

┌─ FinTrack ─────────────────────────────────────────────┐
│  Nuevo gasto: [ Café ] [ $3.500 ] [ Comida ▾ ]  [ + ]  │
│  Filtro: ( Todas )( Comida )( Transporte )( Súper )…    │
├───────────────────────────────┬────────────────────────┤
│  LISTA DE GASTOS              │  RESUMEN                │
│  🍔 Café          $3.500  −+✕ │  Total gastado          │
│  🚗 Uber          $6.200  −+✕ │     $128.400            │
│  🛒 Súper        $42.000  −+✕ │  ─────────────────────  │
│  🎬 Cine          $8.000  −+✕ │  🍔 Comida ████████ 45% │
│                               │  🚗 Transp ████ 22%     │
│  (si no hay: "Sin gastos aún")│  🎬 Ocio   ██ 12%       │
└───────────────────────────────┴────────────────────────┘

🧩 El árbol de componentes (esto es lo importante)

La app NO es un solo archivo gigante. Son 5 componentes, cada uno en su archivo .vue. El padre (App.vue) es el dueño de los datos; los hijos reciben props y emiten eventos.

App.vue · PADRE — dueño del estado (lo integran JUNTOS) │ ├─ FormularioGasto.vue · 👤 A → emite @agregar ├─ PanelResumen.vue · 👤 A ← recibe props :total :por-categoria │ ├─ FiltroCategoria.vue · 👤 B → emite @filtrar └─ ItemGasto.vue (×N) · 👤 B ← recibe prop :gasto → emite @eliminar @ajustar
¿Por qué separar en archivos? Porque cada persona es dueña de sus archivos y así, cuando junten el código en Git, no van a chocar (no habrá conflictos de merge). Esto no es un truco del ejercicio: es la razón real por la que las apps se dividen en componentes.

👥 Cómo se reparten el trabajo — equipo de 2

Trabajo en equipo de verdad: NO se turnan un teclado. Cada persona es dueña de dos componentes y responde por ellos. Al final integran en App.vue y prueban el flujo completo juntos.

👤 Persona A — "Registro y Resumen" (la plata entra y se resume)
  • FormularioGasto.vue → el input donde Cata escribe el gasto y lo emite hacia arriba → Requisitos 1 y 2
  • PanelResumen.vue → recibe el total y el desglose por props y los muestra con barras → Requisitos 5 y 6
En una frase: A hace que el gasto entre y que el resumen se pinte.
👤 Persona B — "Lista y Gestión" (los gastos se ven y se editan)
  • ItemGasto.vue → una fila por gasto; botones para ajustar el monto y eliminar, que emite → Requisitos 3 y 4
  • FiltroCategoria.vue → los chips para filtrar la lista por categoría → Requisito 7
En una frase: B hace que los gastos se listen, se editen y se filtren.

El padre App.vue (Requisitos 5, 6 y 8) lo arman JUNTOS — ahí es donde sus piezas se conectan. Es el punto de integración: sin él, cada componente es una isla.

📜 El contrato de integración — LÉANLO JUNTOS ANTES DE ESCRIBIR CÓDIGO

Este es el momento del trabajo en equipo. Antes de tocar el teclado, los dos acuerdan tres cosas. Si respetan este contrato, sus componentes van a encajar sin tener que arreglar nada al final.

1 · La forma de cada gasto (el objeto que viaja por toda la app)

{ id: 1, descripcion: 'Café', monto: 3500, categoria: 'Comida' }
monto es un número (3500), NO un texto ("3500"). categoria es uno de estos cinco valores exactos: 'Comida', 'Transporte', 'Súper', 'Ocio', 'Otros'.

2 · Los eventos que suben (hijo → padre)

ComponenteEmiteCon qué dato (payload)Quién
FormularioGastoagregar{ descripcion, monto, categoria } (¡sin id! el padre lo pone)A
ItemGastoeliminarid (el número del gasto)B
ItemGastoajustarid, delta (ej: id, 500 o id, -500)B
FiltroCategoriafiltrarcategoria (texto; 'Todas' = ver todo)B

3 · Las props que bajan (padre → hijo)

ComponenteRecibe la propTipoQuién
PanelResumen:totalNumberA
PanelResumen:por-categoriaArray de { nombre, emoji, monto, porcentaje }A
PanelResumen:cantidadNumberA
ItemGasto:gastoObject (la forma de arriba)B
FiltroCategoria:categorias · :activaArray · StringB
Ojo con los nombres. En el <template> las props se escriben en kebab-case (:por-categoria) pero dentro del hijo se declaran en camelCase (porCategoria). Vue traduce solo. Los eventos mantienen su nombre igual en los dos lados (emit('agregar') ↔ @agregar).

🚀 Paso 0 — Montar el proyecto y el repositorio (háganlo JUNTOS, una sola vez)

Una sola persona del equipo hace esto en su computador y comparte el repo. Necesitan Node instalado.

  1. Crear el proyecto Vue + Vite: 
    npm create vite@latest fintrack -- --template vue
cd fintrack
npm install
  2. Dejar los estilos listos. Reemplacen el contenido de src/style.css con el bloque de CSS que está más abajo (sección "Estilos de la app"). Los estilos ya vienen resueltos: ustedes se concentran en la lógica.
  3. Crear la carpeta de componentes: src/components/ y adentro los 4 archivos vacíos: FormularioGasto.vue, PanelResumen.vue, ItemGasto.vue, FiltroCategoria.vue.
  4. Convertirlo en repositorio Git y subirlo a GitHub: 
    git init
git add .
git commit -m "chore: proyecto base FinTrack con Vite + Vue"

# creen un repo vacío en github.com (sin README) y peguen su URL:
git remote add origin https://github.com/USUARIO/fintrack.git
git branch -M main
git push -u origin main
  5. La otra persona clona el repo: 
    git clone https://github.com/USUARIO/fintrack.git
cd fintrack
npm install
  6. Levanten el proyecto con npm run dev y abran la URL que muestra la terminal. Cada vez que guarden, Vite recarga solo. 🔥
Agreguen a la persona que clona como colaborador del repo en GitHub (Settings → Collaborators), o si no, no va a poder hacer push.

🔀 Cómo trabajar en paralelo sin pisarse (flujo Git)

Cada persona trabaja en su propia rama y sobre sus propios archivos. Así el merge es limpio. Este es el flujo que usa cualquier equipo profesional:

  1. Antes de empezar, cada quien crea su rama desde main actualizado: 
    # Persona A
git checkout main && git pull
git checkout -b feat/persona-a

# Persona B
git checkout main && git pull
git checkout -b feat/persona-b
  2. Programan sus componentes, y van guardando avances con commits pequeños: 
    git add .
git commit -m "feat: FormularioGasto emite el evento agregar"
git push -u origin feat/persona-a
  3. Cuando cada uno termina sus componentes, abre un Pull Request en GitHub hacia main. Revísenlos juntos y háganles merge.
  4. La integración final (armar App.vue conectando todo): háganla juntos en una rama feat/integracion, o directo sobre main ya con los dos componentes mezclados. Prueben el flujo completo antes del último push.
¿Por qué casi no habrá conflictos? Porque A toca FormularioGasto.vue y PanelResumen.vue, y B toca ItemGasto.vue y FiltroCategoria.vue. Archivos distintos = Git los mezcla sin problema. El único archivo compartido es App.vue, y por eso lo dejan para el final y lo hacen juntos.

✅ Requisitos

Van por niveles. Terminen el Nivel 1 completo antes de pasar al 2. No salten.

🟢 Nivel 1 — El gasto entra y se muestra (obligatorio)

  1. 👤 A Formulario de gasto. En FormularioGasto.vue: un input de descripción, un input numérico de monto y un <select> de categoría, todos con v-model. (Repaso Lección 3.)
  2. 👤 A Emitir el gasto. Al enviar el <form> (con @submit.prevent), emite el evento agregar con el objeto { descripcion, monto, categoria } y limpia los inputs. No agrega el gasto él mismo: solo avisa al padre.
  3. 👤 B Fila de gasto. En ItemGasto.vue: recibe un gasto por prop (con validación type: Object, required: true) y muestra su emoji de categoría, descripción y monto formateado.
  4. 👥 A+B Pintar la lista. En App.vue: guardan el gasto que llega por @agregar (poniéndole un id nuevo) y renderizan un ItemGasto por cada gasto con v-for y :key. Si la lista está vacía, muestren "Sin gastos aún".

🟡 Nivel 2 — Editar y resumir

  1. 👤 B Eliminar y ajustar. En ItemGasto.vue: un botón ✕ que emite eliminar con el id, y botones − / + que emiten ajustar con (id, -500) / (id, 500). El padre es quien realmente borra o cambia el monto.
  2. 👤 A Panel de resumen. En PanelResumen.vue: recibe por props total, cantidad y porCategoria (con validación) y los pinta: el total grande, y una barra por categoría con su porcentaje.
  3. 👥 A+B Los cálculos (computed) en el padre. En App.vue: total = suma de los montos; porCategoria = arreglo con el monto y porcentaje de cada categoría. Son computed, NO variables que actualizan a mano. El dato manda, la vista lo refleja.

🔴 Nivel 3 — Desafío (si les da el ritmo)

  1. 👤 B Filtro por categoría. En FiltroCategoria.vue: chips (Todas / Comida / Transporte / …) que emiten filtrar con la categoría elegida. El padre filtra la lista mostrada. La categoría activa se marca distinto (recíbanla por la prop activa).
  2. 👤 A Presupuesto con v-model de componente. Un componente MetaMensual.vue con defineModel() para fijar un presupuesto; si el total lo supera, el resumen se pinta en rojo. (Repaso sección 9 de la guía: v-model en componentes.)
  3. 👥 A+B Persistencia. Guarden los gastos en localStorage para que sobrevivan al recargar.

🧠 Conceptos que DEBEN aparecer en su solución

Es la lista de lo que aprendieron. Que no falte ninguno del nivel que alcancen:

  • defineProps con validación (type / required / default) — en ItemGasto y PanelResumen
  • defineEmits — declarar los eventos de cada hijo
  • emit con payload — objeto (agregar) e id (eliminar)
  • v-for + :key pasando datos por props (padre → ItemGasto)
  • Comunicación entre hermanos vía el padre: FormularioGasto → App → PanelResumen
  • computed — para el total y el desglose (¡no lo calculen a mano!)
  • v-model — en los inputs del formulario (repaso Lección 3)
  • @submit.prevent — para enviar sin recargar (repaso Día 1)
  • Git en equipo — ramas, commits, push y merge en un mismo repo
  • (Nivel 3) defineModel — v-model en un componente propio

💡 Pistas técnicas — no es la solución, es el mapa

  • El dato vive en el padre. El array gastos es un ref en App.vue. Los hijos NO tienen su propia copia: reciben lo que necesitan por props.
  • El hijo nunca modifica una prop. ItemGasto no borra ni cambia el gasto: emite y deja que el padre —el dueño— decida. Si intentan mutar la prop, Vue les grita por consola.
  • El total es computed. const total = computed(() => gastos.value.reduce((s, g) => s + g.monto, 0)). Si lo guardan en una variable y lo suman a mano, tarde o temprano se desincroniza.
  • Para porCategoria: recorran las 5 categorías, sumen los montos de los gastos de cada una, y el porcentaje es monto / total * 100. Devuelvan un arreglo de objetos.
  • El monto del input llega como texto. Usen v-model.number o conviertan con Number(...) antes de emitir.
  • El id lo pone el padre, no el formulario. El hijo emite el gasto sin id; el padre hace { id: ++idActual, ...datos }.

🏁 ¿Cuándo está "listo"?

Su FinTrack está listo cuando pueden hacer esta demo entre los dos, sin tocar código:

  1. Escribir un gasto en el formulario (Persona A) y verlo aparecer en la lista de la izquierda.
  2. El total de la derecha cuadra con la suma real y se actualiza solo.
  3. Los botones − / + cambian el monto (Persona B) y el resumen reacciona al instante.
  4. ✕ elimina un gasto y las barras del resumen se recalculan.
  5. Los chips de filtro muestran solo la categoría elegida (Nivel 3).
  6. Todo el código está en un solo repositorio, con commits de los dos integrantes en el historial.
Si los 6 pasos funcionan: felicitaciones, construyeron una app de finanzas con componentes que se comunican, y la armaron como un equipo real. 🎉
Ojo: los archivos .vue no se abren con doble clic en el navegador — necesitan el proyecto Vite corriendo (npm run dev). Lo único que se publica en GitHub Pages es esta página de instrucciones.
